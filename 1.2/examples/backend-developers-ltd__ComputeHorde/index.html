subnet:

task:
  summary: ComputeHorde is Bittensor's decentralized GPU-compute subnet (netuid 12). Miners run "executors"
    (dockerized GPU instances of declared hardware classes, e.g. spin_up-4min.gpu-24gb) that perform compute
    jobs. Validators earn per-block "allowance" (executor-seconds, proportional to executor count and
    validator stake) and spend allowance blocks to pay miners for ORGANIC jobs forwarded from other subnets'
    validators via a Facilitator. The current scoring rewards miners by the executor-seconds of finished
    organic jobs that were properly paid for with valid allowance blocks. SYNTHETIC jobs (validator-generated)
    test executor availability and hardware authenticity; organic results are cross-validated against
    a Trusted Miner. Scores per coldkey are split across hotkeys (main hotkey gets the largest share),
    with a "dancing" bonus for changing the main hotkey between cycles to discourage weight-copying.
  submission_format:
  - compute_service
scoring_signals:
- name: allowance_paid_executor_seconds
  metric_kind: throughput
  direction: higher_is_better
  weight: 1.0
  normalization: other
  notes: 'Per-executor-class batch normalization: score/total * executor_class_weight. calculate_allowance_paid_job_scores
    gives each miner score equal to executor_seconds_cost of organic jobs that finished in the cycle and
    were validated as paid with fresh, unspent, sufficient allowance blocks (scores[executor_class][miner]
    += job_cost). Invalid spendings (CannotSpend / ErrorWhileSpending) score nothing.'
- name: organic_job_completion
  metric_kind: other
  metric_kind_other: fixed score per successfully completed, non-cheated organic job
  direction: higher_is_better
  weight: null
  normalization: none
  notes: 'Legacy/alternative organic scoring in calculations.py (score_organic_jobs / calculate_organic_scores):
    each COMPLETED, not-cheated, not-on-trusted-miner organic job contributes DYNAMIC_ORGANIC_JOB_SCORE
    (default 1.0). The README''s allowance-based engine is the active path; this fixed-per-job path is
    documented in the scoring README as the organic-job contribution.'
ground_truth_sources:
- kind: organic_traffic
  description: Real user/cross-subnet jobs forwarded into ComputeHorde via the Facilitator gateway; completing
    these paid organic jobs is the primary scored activity.
  source_identifier: Facilitator (organic requests from other subnets' validators)
  trust_model: adversarial
- kind: reference_model
  description: A separate GPU 'Trusted Miner' (same code as a regular miner, not registered in the metagraph,

    results, against which miners' organic results are cross-checked for cheating.
  source_identifier: Trusted Miner
  trust_model: oracle
- kind: hardware_probe
  description: Synthetic validator-generated jobs benchmark/test executors to confirm declared hardware
    classes and availability; jobs are matched to advertised hardware (e.g. A6000 for A6000-required tasks).
  source_identifier: synthetic jobs (SyntheticJobBatch)
  trust_model: adversarial
aggregation:
  method: weighted_average
  normalization: other
  decay_rate: null
  min_weight_floor: null
  burn_allocation:
    enabled: true
    address_or_uid: null
    fraction: null
    notes: 'apply_dancing_burners directs DYNAMIC_BURN_RATE fraction of incentive to DYNAMIC_BURN_TARGET_SS58ADDRESSES
      (weights *= (1 - burn_rate)), with DYNAMIC_BURN_PARTITION controlling the primary burner''s share
      when multiple are registered. Defaults: DYNAMIC_BURN_RATE = 0.0 and DYNAMIC_BURN_PARTITION = 0.0
      (burning disabled by default; activated via dynamic config).'
  notes: 'Normalization (other): per executor class, (miner_score / total_executor_class_score) * executor_class_weight,
    summed across executor classes per hotkey; final on-chain weights produced by bittensor.utils.weight_utils.process_weights
    with min_allowed_weights / max_weight_limit. Executor-class weights default DYNAMIC_EXECUTOR_CLASS_WEIGHTS
    = "spin_up-4min.gpu-24gb=98,always_on.llm.a6000=1,always_on.test=0,always_on.cpu.8c.16gb=1". Dancing
    bonus multiplier (1 + DYNAMIC_DANCING_BONUS, default 0.1) applied to a coldkey total when its main
    hotkey changed vs the previous cycle; main hotkey then gets MAIN_HOTKEY_SHARE (default 0.8) of the
    coldkey score, others split the remainder.'
weight_setting:
  cadence: per_epoch
  method: Celery task set_scores scores the current cycle (722-block cycles), normalizes per executor
    class, applies dancing burners, maps hotkeys->uids, and submits weights via the Pylon client (p_client.identity.put_weights).
    Commit-reveal gating via CommitRevealInterval when enabled.
  on_chain_call: commit_reveal
  tempo_or_interval: 722 blocks (DYNAMIC_COMMIT_REVEAL_WEIGHTS_INTERVAL; cycle length)
anti_gaming:
- kind: commit_reveal
  enforcement: barrier
  description: 'Commit-Reveal weights: validators post hidden weights and reveal them in the next epoch,
    making copying of current weights impossible. DYNAMIC_COMMIT_REVEAL_WEIGHTS_ENABLED default True,
    DYNAMIC_COMMIT_REVEAL_WEIGHTS_INTERVAL default 722 blocks; commit/reveal windows computed by CommitRevealInterval.'
- kind: other
  kind_other: executor dancing (random GPU reallocation across UIDs + dancing bonus)
  enforcement: other
  description: 'Executor Dancing: miners move GPUs across multiple UIDs and change their declared main
    hotkey between cycles; changing it triggers a 10% (DYNAMIC_DANCING_BONUS) boost, encouraging variance
    to further reduce effectiveness of copying old weights.'
- kind: collateral
  enforcement: penalty
  description: Validators can require miners to deposit collateral (DYNAMIC_MINIMUM_COLLATERAL_AMOUNT_WEI,
    default 1e16 wei = 0.01 TAO) to be eligible for organic jobs. Cheating miners are slashed (DYNAMIC_COLLATERAL_SLASH_AMOUNT_WEI,
    default 5e15 wei = 0.005 TAO) via the collateral contract.
- kind: hardware_validation
  enforcement: rejection
  description: Jobs are matched to advertised hardware (executor classes); synthetic jobs test executor
    availability and hardware authenticity, and unconfigured executor classes are excluded from scoring
    (weight defaults to 0.0).
- kind: deterministic_check
  enforcement: rejection
  description: Organic results are cross-validated against a Trusted Miner that pre-runs validation tasks
    to establish expected results; jobs flagged cheated (job.cheated) are excluded from scoring and trigger
    temporary blacklisting (DYNAMIC_JOB_CHEATED_BLACKLIST_TIME_SECONDS, default 4 hours).
- kind: rate_limit
  enforcement: barrier
  description: Allowance is minted per miner-validator-executorClass per block and validators must spend
    valid, fresh, unspent, sufficient allowance blocks to pay for a job; overspending, stale, or double-spent
    blocks void the incentive (CannotSpend), enforcing fair usage spread across miners.
